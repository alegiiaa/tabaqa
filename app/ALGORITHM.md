# The Tabaqa Algorithm — end to end, with the judge Q&A

> **Purpose of this document:** the single thing to study before the pitch.
> Every number in it is generated by code in this repo and can be reproduced
> live. Sections 1–7 are the methodology; **section 8 is the hard-question
> drill** for a SAMA-, SIMAH- or D360-literate judge.

---

## 0. The 30-second answer ("so what is your algorithm?")

*"Tabaqa ingests a person's **full cash-flow reality** — any bank statement plus
any wallet export, in any format — normalizes it into one canonical ledger,
de-duplicates internal transfers, verifies income against Masdr ground truth in
three honesty tiers, and computes **seven cash-flow features**. A transparent
additive scorecard maps those features to a 1–99 score where **every point has a
reason code**, then to a probability of default and a SAMA-compliant
affordability decision. The features are **validated on real defaults**: on 682
real loan accounts, adding the wallet layer lifts AUC from 0.66 to 0.86 — and on
thin-file borrowers, from coin-flip 0.60 to 0.78."*

---

## 1. Pipeline map (what runs on every applicant)

```
any export ──▶ [0] universal adapter ──▶ [1] clean ──▶ [2] enrich ──▶ [3] reconcile
 (CSV/file)     web/src/lib/adapters.ts    clean.py      enrich.py      reconcile.py
                                                                            │
   decision ◀── [7] affordability ◀── [6] score ◀── [5] features ◀── [4] verify + PFC
                sama.py               scoring/       features.py     verify.py · pfc.py
                                      scorecard.py
```

| # | Stage | File | What it does |
|---|---|---|---|
| 0 | **Universal adapter** | `web/src/lib/adapters.ts` | Any bank/wallet export → canonical rows: Arabic/English headers, preambles, Arabic-Indic digits, DD/MM vs MM/DD vs Hijri dates, debit/credit vs signed vs type-column amounts, per-file institution fingerprinting (`bank:alinma`, `wallet:urpay`) |
| 1 | Clean | `pipeline/clean.py` | Normalize Arabic text/digits in descriptions |
| 2 | Enrich | `pipeline/enrich.py` | Merchant + category labeling (rules first, LLM long-tail assist) |
| 3 | **Reconcile** | `pipeline/reconcile.py` | Match bank→wallet transfers (equal amount ±0.01 SAR, ≤3 days, transfer keywords) and tag both legs `internal_movement` — **income is never double-counted across sources** |
| 4 | **Verify** | `pipeline/verify.py` | 3-tier provenance against Masdr: `amount_verified` (payslip wage + salary-IBAN match) > `source_verified` (registered gig establishment) > `inferred`. Also stamps the Plaid PFC taxonomy (`pfc.py`) |
| 5 | Features | `pipeline/features.py` | The 7 cash-flow features (below) |
| 6 | Score | `scoring/scorecard.py` | Additive points → 1–99 score → PD → reason codes |
| 7 | Affordability | `sama.py` | SAMA responsible-lending caps → max installment / decision |

The pipeline consumes one canonical shape (`pipeline/schema.py`): every
transaction is `source`-tagged (`bank:<id>` / `wallet:<id>`), so **multi-source
fusion is the data model**, not a feature bolted on.

## 2. The seven features (and the proof each carries signal)

Computed in `pipeline/features.py` from the reconciled, verified ledger.
Information Value (IV) is measured against **real defaults** on Berka
(`eval/berka_train.py` → `eval/DATA_REPORT.md`); IV ≥ 0.1 = predictive,
≥ 0.3 = strong.

| Feature | Definition | IV (real defaults) |
|---|---|---|
| `balance_volatility` | Coefficient of variation of running balance | **1.295** (strongest) |
| `income_regularity` | 1 − CV of monthly income, × months-coverage | **0.442** |
| `recurring_obligation_load` | Monthly loan/rent obligations ÷ income | **0.318** |
| `min_balance` | Minimum of the event-by-event running balance | **0.317** |
| `income_expense_ratio` | Total inflow ÷ total outflow | **0.300** |
| `avg_balance` | Mean running balance | **0.244** |
| `nsf_count` | Times any account went negative | 0.000 on Berka* |
| `verified_income_share` | Share of income that is Masdr-verified | (Tabaqa-specific — no Berka analogue) |

\* Berka accounts never overdraft, so NSF *cannot* show signal there. We report
that honestly and keep the feature because Saudi wallet/bank data does have
NSF events. Saying this unprompted reads as rigor, not weakness.

## 3. The scorecard (the live model — memorize its shape)

`scoring/scorecard.py` — a **transparent additive points model**, the exact form
`optbinning.Scorecard` produces, kept dependency-free. Base = **20 points**,
then first-matching bin per feature:

| Feature | Bins → points |
|---|---|
| Income regularity | ≥0.8 → **+18** · ≥0.6 → +12 · ≥0.4 → +6 · else **−6** |
| Verified income share | ≥0.7 → **+14** · ≥0.4 → +8 · else 0 |
| NSF events | 0 → **+12** · ≤2 → +3 · else **−12** |
| Income/expense ratio | ≥1.4 → +8 · ≥1.15 → +5 · ≥1.0 → +1 · else **−8** |
| Min balance | ≥1000 → +6 · ≥0 → +2 · negative → **−8** |
| Balance volatility | ≤0.4 → +4 · ≤0.8 → +1 · else −4 |
| Obligation load | ≤0.3 → 0 · ≤0.5 → −5 · else **−12** |

`score = clamp(20 + Σ points, 1, 99)`. Every non-zero bin emits a
**reason code** (e.g. `regular_income`, `zero_nsf`); the top-3 by |points| are
the API headline. Explainability isn't a layer on top — **the model is its own
explanation**, which is what an adverse-action notice needs.

**PD transform:** `pd = clamp(1.39 · (1 − score/99)², 0.002, 0.99)` — a smooth
convex map anchored so the worst score ≈ the observed worst-band default rate
(~39% on Berka band 1) and calibrated against the band table below. Risk flag:
low < 6% PD, medium < 15%, else high.

**The three-model separation (a sharp judge probes this — know it cold):**

| Model | What | Where | Role |
|---|---|---|---|
| Live MVP scorer | Expert-weighted additive points (above) | `scoring/scorecard.py` | Runs the demo; fully explainable |
| Validation model | Fitted logistic regression, same features | `eval/berka_train.py` | Proves the features predict real defaults |
| Ablation baseline | Fitted model on bureau-view features only | `eval/ablation.py` | The thing we switch off to price the wallet layer |

The claim is **never** "our demo scorecard achieved AUC 0.89." The claim is:
*"the same seven features, when fitted, achieve it — and production swaps in a
WOE-binned `optbinning.Scorecard` behind the identical I/O contract."*

## 4. The evidence stack (all reproducible from this repo)

**Dataset:** Berka / PKDD'99 (CTU public MariaDB) — the only public dataset
pairing real transaction histories with real loan outcomes. **682 loan accounts
with strictly pre-loan history · 76 defaults · 11.1% bad rate.** No leakage:
features are computed only on transactions dated before the loan.

| Experiment | File | Headline |
|---|---|---|
| **Feature validation** | `eval/berka_train.py` → `DATA_REPORT.md` | Holdout **AUC 0.890 / KS 0.683**; 5-fold CV **0.858 / 0.562**; monotonic bands **38.7% → 10.3% → 2.9% → 3.7% → 0.0%** |
| **Wallet-layer ablation** | `eval/ablation.py` → `ABLATION.md`, `model_card.json` | Bureau view **0.661** → + wallet **0.864**; lift **+0.203**, 95% CI **+0.144…+0.268**, >0 in **100% of 3,000 bootstraps**; Brier **0.2245 → 0.1262** |
| **Thin-file conditional** | same | Shortest-history third (252 accounts, ≤11 months): bureau **0.596** (≈ coin-flip) → **0.775** |
| **Swap-set** | same | At fixed 60% approval: realized defaults **7.6% → 2.9% (−61%)**; 100 wallet-rescued borrowers default at **3.0%**, 100 wallet-rejected (bureau-approved) at **22.0%** |
| **Synthetic corpus + TSTR** | `eval/corpus.py`, `corpus_train.py` | Gaussian-copula corpus of **1,000,000 accounts** learned from the 682 real ones (191,556 real transactions); **train-on-synthetic/test-on-REAL retains 96%** of real-data AUC (0.870 vs 0.904) |
| **Enrichment accuracy** | `eval/eval_enrich.py` + `labeled_transactions.jsonl` | Categorization measured against a hand-labeled transaction set; regression-guarded (`eval_regression.py`) |

The in-app **Model validation** panel (`ModelCardPanel.tsx`) renders all of this
live, including the bureau-view ↔ wallet-layer toggle.

## 5. The verification layer (why the data can be trusted)

Three tiers, assigned per transaction, never inflated:

1. **`amount_verified`** — salary rows whose counterparty IBAN matches the
   Masdr payslip employer IBAN *and* amount matches the registered wage.
2. **`source_verified`** — gig income from platforms registered as
   establishments (e.g. Jahez, HungerStation) — the *source* is real even if
   monthly amounts vary.
3. **`inferred`** — everything else (P2P transfers stay here **by design**;
   we never Masdr-stamp a P2P row).

`verified_income_share` feeds the scorecard (+14 max) — so verification isn't
UI decoration; it's priced into the score.

---

## 6. The regulatory layer (`sama.py`)

Per SAMA's responsible-lending framework (Circular 46538/99): binding
salary-deduction cap **33.33%** for employees, **25%** for pensioners, plus the
**total-obligations ceiling** by income band (45% / 55% / 65% incl. real-estate
finance). `/v1/affordability` returns the binding cap, the max monthly
installment, and the decision — configurable to a lender's internal policy.
Consent-based, read-only account access (AIS); no payment initiation.

## 7. Universal ingestion (the open-banking claim, made concrete)

The adapter (`web/src/lib/adapters.ts`) makes "works with any platform" literal:

- **Format detection**, not format requirements: header-synonym matching in
  Arabic + English, preamble skipping, delimiter sniffing (`,` `;` tab `|`),
  Arabic-Indic digit normalization, DD/MM ↔ MM/DD resolution across the whole
  file, textual + Arabic month names, **Hijri→Gregorian** conversion (tabular
  calendar, ±1 day — immaterial for monthly features).
- **Amount conventions:** signed · debit/credit columns · amount+type column
  (دائن/مدين, CR/DR) · `(500)` and trailing-minus negatives · `ر.س`/SAR tokens ·
  `١٬٠٠٠`-style separators.
- **Institution fingerprints:** one declarative entry per bank/wallet
  (filename + preamble matching → `bank:alinma`, `wallet:urpay`, …, incl.
  D360). **Adding an institution = adding one profile line.**
- **Fusion:** each file resolves to its own `source`; rows merge into one
  ledger; `reconcile` de-duplicates cross-source transfers downstream.

**Live proof (demo: Connect → My data → "Load Alinma + urpay exports"):** an
Arabic Alinma e-statement (Arabic headers, مدين/دائن columns, Arabic-Indic
digits, running balance) + an English urpay export (textual dates, type-signed
amounts) → one ledger, two source-tagged accounts, **score 82, bank-only 4,000
→ true 10,000 SAR (+6,000 revealed)** — identical economics to the scripted
demo, but computed from two raw files in different formats.

## 8. The hard-question drill (SAMA / SIMAH / D360 judge)

**Data & validation**

- **"Berka is Czech data from the 1990s. Why does it transfer to Saudi?"**
  It doesn't need to transfer coefficients — it validates *feature-outcome
  mechanics*. Berka is the only public dataset with real transactions *and*
  real defaults; no Saudi open dataset has either. The load-bearing claim is
  that the same seven features produce a monotonic, out-of-sample default
  ranking on real outcomes — not that 0.89 is the Saudi AUC. Production
  refits on the lender's own AIS history behind the same I/O contract.

- **"AUC 0.89 is suspiciously high for cash-flow-only."**
  Agreed — typical cash-flow-only models sit at 0.65–0.75 (FinRegLab). Berka
  is a separable benchmark, we say so in `DATA_REPORT.md`, and we lead with
  the 5-fold CV number (0.858) not the holdout. The defensible number is the
  *controlled lift*: +0.203 AUC over a bureau baseline **on the same
  accounts**, CI +0.144…+0.268.

- **"Is the score in your demo the trained model?"**
  No, and we keep them deliberately separate (§3): the demo runs a transparent
  expert-weighted scorecard so every point is attributable; the fitted model
  exists to prove the features predict. Same features, same I/O contract —
  production swaps in the WOE-binned scorecard. Mixing them up is the trap;
  we don't.

- **"You validated on approved loans only. Reject inference?"**
  Correct — labels exist only for booked loans, so measured lift is on the
  booked population. It's a named caveat on the model card. Production adds
  standard reject-inference (parceling / augmentation) on the lender's
  through-the-door population. Note the swap-set already shows the
  *direction*: bureau-approved-but-wallet-rejected borrowers default at 22%.

- **"Are your PDs calibrated, or just a ranking?"**
  Both measured: Brier improves 0.2245 → 0.1262 with the wallet layer and the
  reliability curve is on the model card. The demo's PD transform is anchored
  to the observed band default rates (worst band ≈ 39%).

- **"Why is `nsf_count` in the model if its IV is zero?"**
  Zero *on Berka* — Czech accounts in this dataset never overdraft, so the
  feature can't fire. It stays because NSF is a classic cash-flow default
  signal and Saudi wallet data does contain it. We report the 0.000 rather
  than hiding the row — that's the honesty tell.

**Model design**

- **"Why these features and not a deep model over raw transactions?"**
  Because a credit decision must produce adverse-action reasons. Seven
  interpretable features + additive points = every decision decomposes into
  reason codes. The ablation proves the features capture the signal that
  matters (+0.203); a black box would add marginal AUC at the cost of
  explainability SAMA-supervised lenders can't accept.

- **"How do you prevent gaming — someone routing money in circles?"**
  Reconciliation runs before features: bank↔wallet transfers matched on
  amount (±0.01 SAR) within 3 days with transfer keywords are tagged
  `internal_movement` and excluded from income and spend. Self-transfers
  therefore add zero income. And unverified inflows cap the score:
  `verified_income_share` only rises via Masdr-checked salary or registered
  gig platforms — P2P stays `inferred` and earns no verification points.

- **"What about fairness/bias?"**
  The Tabaqa model uses **zero demographic features** — no age, gender,
  nationality, or district. Behavior-only. (The *bureau baseline* in the
  ablation contains demographics precisely because that's what today's
  proxies look like — the thing we outperform.)

**Data engineering & scale**

- **"You claim 1M accounts — is your validation circular?"**
  The 1M corpus is a Gaussian-copula generator *learned from* the 682 real
  accounts, used for scale/segment demos — **never** for validation claims.
  The anti-circularity check runs the honest direction: train on synthetic,
  **test on real** (TSTR) → 96% of real-data AUC retained (0.870 vs 0.904).
  All headline metrics come from real accounts only.

- **"Any platform? What does adding bank X actually take?"**
  One declarative profile in the adapter (fingerprint + any header synonyms
  not already covered). The canonical schema, reconciliation, verification,
  features and scoring are source-agnostic — demoed live by fusing an Arabic
  Alinma e-statement with an English urpay export (§7).

- **"CSV uploads can be doctored."**
  Yes — the CSV path is the demo's stand-in for consented AIS pulls. The
  trust boundary is the verification layer: Masdr payslip/IBAN and registered
  establishments verify the income that carries score weight. In production,
  data arrives signed from the open-banking APIs (Tarabut/Lean-style AIS),
  not from user files.

**Regulatory & deployment**

- **"Do you need a SAMA license?"**
  Deployment route: data processor inside the lender, on customer consent,
  read-only AIS, no payment initiation (no PIS) — the assessment layer, not a
  lender or bureau. Wallet access is gated on partnership/consent and demoed
  on consented/simulated data; we say that plainly.

- **"How is this different from SIMAH?"**
  SIMAH scores *credit history* — it's structurally blind to people without
  one. Tabaqa scores *cash-flow reality*, which exists for every gig worker
  with a wallet. The ablation is that difference as a number: bureau-view AUC
  0.596 on thin-file borrowers (noise) vs 0.775 with the wallet layer. We
  complement the bureau; we don't replace it.

- **"Your DBR numbers?"**
  Base salary-deduction cap 33.33% salaried / 25% pensioner; total-obligation
  ceilings 45% (and up to 55–65% including real-estate finance by income
  band), per SAMA's responsible-lending circular — implemented in `sama.py`
  and configurable to lender policy.

**The one-liner to land:** *"Most teams will show you a model. We're showing
you a controlled experiment on real defaults, a verification layer that prices
honesty into the score, and an ingestion layer that reads whatever a Saudi
bank or wallet actually exports."*

## 9. Known limitations (say them before they're asked)

1. Berka is a 1990s Czech proxy — the only public real-transactions +
   real-defaults data; Saudi refit is the production path.
2. Labels are booked loans only (no reject inference on a public benchmark).
3. The live demo scorer is expert-weighted; the fitted scorecard swap is
   scaffolded (`scoring/train.py`), not shipped in the demo.
4. CSV upload is the demo's stand-in for signed AIS feeds.
5. Hijri conversion is tabular (±1 day) — immaterial at monthly granularity.
6. The bureau baseline is a demographics/district/loan-terms proxy (Berka has
   no bureau feed) — which makes the measured lift *conservative*.
